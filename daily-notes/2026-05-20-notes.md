# OOP Basics

## abstraction
  - exposing essential behavior while hiding implementation complexity.
  - Usually achieved using: interfaces, abstract classes, APIs
## encapsulation: 
  - protecting internal state/data and controlling access.
  - Usually achieved using: private fields, setters/getters, controlled methods
## Overriding (Runtime Polymorphism)
  - Same method signature But: 
    - Access Modifier: Lower down **NOT allowed** (widening allowed)
    - return types: Lower down **allowed** (co-variant or child types) 
    - throws clause: Lower down **allowed** (child types)
  - Can we override `static` methods ? 
    - **No** (But NO CE, No RE), It becomes "method hiding"
  - Can we override `private` methods ?
    - **No**, it's not visible to child class (But NO CE/RE).
    - It becomes private methods of that child class. 
  - Can we override `final` methods ?
    - **No**. final means constant no change, Whereas overriding means change. 
    - Hence, both combination (final & override) can't co-exist.

# java.lang 
**(String, StringBuilder, StringBuffer, immutability, thread-safety, constant Pool)**

## Strings: 
  - Why immutability ?
    1. **Security**: Strings used in: DB, URLs passwords, file paths, ...etc. which CAN NOT change unexpectedly.
    2. **Thread safety**: Immutable objects are naturally thread-safe.
    3. **String Pool optimization**: SCP. Frequently used strings can be reused. Saves COST OF OBJECT CREATION/DESTRUCTION for JVM
    4. **Hashcode caching**: Strings are heavily used in HashMap keys. Immutability guarantees stable hashcodes.
## StringBuilder vs StringBuffer
- StringBuilder: non-thread-safe | general purpose string manipulation object (BUT NOT in multithreading) | data inconsistency(UNSAFE): in concurrent thread modification. | relatively faster due to **asynchronous** methods.
- StringBuffer: thread-safe (synchronized) | designed for multithreaded env. | No data inconsistency(UNSAFE): in concurrent thread modification. | relatively Slower due to **synchronous** methods.
- Similarity: mutable | concurrent thread modification on a particular string object allowed  |Hence Better update performance than immutable objects |
- When would you use each? 
  - Use StringBuffer only in case of Multithreaded env. remaining all places use StringBuilder.  
## difference b/w `==` and `equals()` for Strings 
  - `==` reference comparison 
  - `equals()` content comparison
  - `String a = new String("hello");
    String b = new String("hello"); 
    System.out.println(a == b);       // false
    System.out.println(a.equals(b));  // true` 
## equals() & hashCode()
- if `a.equals(b) == true` then `a.hashCode() == b.hashCode()` MUST ALSO BE TRUE
- Otherwise, collections like: HashMap, HashSet --> behave incorrectly.

# Collections & Map

## ArrayList vs LinkedList 
  when to choose each?
  - ArrayList: Fast read/iterate/search operations. RandomAccess | Mostly used.
  - LinkedList: For update(insert/delete) operations. 
## HashSet vs TreeSet (when to choose each?)
  - HashSet: NO sorting | UDS: Hashtable | complexity : O(1) 
  - TreeSet: sorting & navigation support | UDS: Red-Black (balanced) tree | complexity: O(logN) 
## HashMap vs ConcurrentHashMap
- HashMap: NOT thread-safe | Multiple threads modifying it can cause **data inconsistency** 
- ConcurrentHashMap: Thread-safe | Allows concurrent reads/writes efficiently | Used in multithreaded environments
## HashMap vs LinkedHashMap
- HashMap: no-ordering | complexity: Hence performance up i.e. O(1) | UDS: Hashtable | faster read/search.
- LinkedHashMap: ordered | complexity: Hence performance little down | extends HashMap | UDS: Hashtable + DLL | Good for more insert/delete operations.
- similarity: key-value format | no duplicate | 1 null key entry allowed 
- summary: insertion-order is a main difference.

## Practical Engineering 
1) Suppose: A user registration API is getting duplicate email entries sometimes. What could be 2–3 possible causes?
  - **Unique constraint checks at DB level**. Like: `UNIQUE(email)`.
  - This is the REAL final protection. Because application checks alone are not enough.
- Q) Why application-level check alone can fail?
  - In case of **race condition** in multithreaded environment. 
  - Two requests arrive simultaneously:
    `Request A -> checks email -> not found
    Request B -> checks email -> not found`
  Both then insert.
  - Result: duplicate email.
  - this is called RACE CONDITION.
- Q) What should be Correct production approach ?
  - **Application validation** checks: empty email, valid format, existing user.
  - PLUS **Database unique constraint** : Final guarantee.
- **Additional causes of duplicates:** 
  - Missing transaction handling.
  - Distributed systems issue: Multiple services inserting inconsistently.
  - Event retries/message duplication In async systems: (Kafka retries, duplicate event processing)
- **Summary -> Think around**: DB unique constraints on email field, application level validation especially under concurrent requests flow,  transaction handling, logs, concurrent registration flows to identify race conditions.
2) Suppose: An API suddenly becomes very slow in production. What are the first things you would investigate?
- A. First identify --> is problem: 
  - application-wide? | one endpoint? | one region? | one DB query?
- B. Then check --> Infrastructure: 
  - CPU | Memory | thread pool | GC Pause | 
- C. Then check --> dependencies
  - DB latency | downstream API latency | network timeout | retries
- D. Then check --> logs/metrics
  - response time | error spikes | tracing | slow queries
